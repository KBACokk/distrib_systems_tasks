# Вопросы для самопроверки

1. Что такое Server Streaming? Приведите пример использования.

Файл proto испольуется как единый контракт. В нем описываются структуры данных и методы сервиса (независимо от языка программирования). Мы не пишем классы сообщений сразу на Python, потому что если писать их в proto, то мы можем автоматически генерировать из для разных языков, что упрощает согласованность системы

2. Чем Server Streaming отличается от просто получения большого списка (List) в одном ответе?

Unary RPC - это простейший тип взаимодействия в grpc, при котором клиент отправляет 1 запрос и получает 1 ответ. Он похож на REST тем, что в нем клиент тоже делает запрос и ожидает ответ

3. Какие метрики производительности существуют (Latency, Throughput, CPU Usage)?

gRPC использует Protocol Buffers, который работает быстрее и занимает меньше места, чем JSON. Так же gRPC построен поверх http2, что позволяет использовать мультиплексирование, сжатие заголовков и тд

4. Почему важно переиспользовать канал (Channel) в gRPC клиенте при замерах производительности?

Обратная совместимость в protobuf - это возможность обновлять структуру данных так, чтобы старые и новые версии клиентов и серверов продолжали работать друг с другом. Номера полей нужны для идентификации полей при сериализации и десериализации данных

5. В каких сценариях REST может оказаться быстрее или удобнее gRPC?

Stub - это сгенерированный не клиенте объект, через который вызываетсю удаленные методы сервиса, скрывая детали сетевого взаимодействия. Servicer - это базовый класс на сервера, который нужно реализовать, чтобы обрабатывать вызовы клиентов


6. Что такое "Head-of-Line Blocking" и как HTTP/2 (и gRPC) решает эту проблему?

REST для публичных API (простота, совместимость с браузерми, удобство для разработчиков), а gRPC для внутренних микросервисов (производительность)
